# 一、开发

## 1、单元测试

```
## 背景信息

作为 Java 架构师，需为以下两个类生成对应的单元测试类：

- `com.demo1`
- `com.demo2`

目标测试类必须存放在标准 Maven 测试目录下：

- `src/test/java/com/jd/merchant/business/platform/core/service/filetask/impl`

## 要求与规范

### 1. 命名规范
- 所有测试方法必须使用驼峰命名法（如 `testExportSuccessWithValidData`），禁止使用下划线。

### 2. 测试框架与注解
- 测试类必须使用 `@ExtendWith(MockitoExtension.class)` 作为运行器。
- 被测服务类使用 `@InjectMocks` 注解注入。
- 所有**外部依赖**（如 Service、Repository、Util 等）使用 `@Mock` 注解模拟。
- 每个测试方法必须使用 `@DisplayName` 注解，并用**中文明确描述测试场景**（例如：`@DisplayName("导出成功 - 输入有效业务数据")`）。

### 3. 数据与 Mock 管理
- 所有 Mock 行为和测试数据必须在**每个测试方法内部定义**，禁止使用 `@BeforeEach`、`@Before` 或任何全局初始化方法。
- 必须覆盖以下逻辑分支：
  - 正常成功路径（含典型业务数据）
  - 异常路径（如服务抛出业务异常、系统异常）
  - 边界条件（如空输入、空集合、单元素集合、超大集合）
  - 条件分支（每个 `if/else`、`switch` 分支）
  - 异常处理（每个 `try/catch` 块的 catch 路径）
- **不得 Mock 被测类自身的任何方法**（仅允许 Mock 其依赖的外部组件）。

### 4. 依赖合法性
- 生成的代码中引用的所有类、字段、方法必须确保在项目当前模块或 Maven 依赖中真实存在，禁止引入不存在的类或过时 API。
- 每个测试方法上方必须包含独立注释块，说明：
  - 测试目的
  - 覆盖的具体逻辑分支
  - 预期行为（如返回值、异常类型、调用次数等）
- 禁止使用行尾注释（即 `// 注释` 写在代码行末），所有注释必须独立成行并置于相关代码上方。

### 5. 集合初始化规范
- 所有集合（如 `List`、`Set`、`Map`）初始化时必须显式指定**最小合理初始容量**（例如：已知将添加 3 个元素，则使用 `new ArrayList<>(3)`；若集合为空，则使用 `new ArrayList<>(0)`）。

### 6. 其他限制
- 仅允许创建上述两个类对应的单元测试类（文件名需符合 JUnit 规范，如 `DepositInvoiceDetailExportHandlerTest.java`）。
- 不得修改任何业务逻辑代码。
- 不得额外创建工具类、配置类、辅助类或测试基类。
```



## 2、代码注释

```
请对以下 Java 代码进行深度美化与注释增强，严格遵循以下六点要求：

1. **优化类、方法、成员变量的注释**  
   - 为类添加详细的  `Javadoc` 注释，说明其核心职责、所属模块及关键行为。  
   - 为每个 `public/protected` 方法添加完整的 `Javadoc`，包含 `@param`、`@return`、`@throws`（如适用），并描述业务语义和使用场景。  
   - 成员变量若具有业务含义，可在字段上方统一加一段注释说明其作用，**禁止使用尾行注释（即行末 // 注释）**，避免逐字段重复注释。
2. **为方法内部代码添加分层注释**  
   - 在方法体内使用结构化层级注释（例如：`// 1. 校验输入参数合法性`、`// 2. 查询关联业务数据`、`// 3. 构建返回结果并记录日志`），清晰划分逻辑阶段。  
   - 分层注释必须体现“准备 → 处理 → 验证/返回”或类似业务流程，提升可读性与可维护性。
3. **优化变量命名，提高可读性**  
   - 所有局部变量、参数、返回值命名必须具有明确业务语义，禁止使用 `data`、`obj`、`tmp`、`res`、`flag` 等模糊词汇。  
   - 优先使用名词短语（如 `merchantLabelRule`）或动宾结构（如 `buildResponseDto`），确保命名自解释。
5. **检查并修复代码中的格式问题**  
   - 修复括号不匹配、缺少换行、异常吞没、魔法值未提取等问题（仅限格式与可读性层面，不改变逻辑）。  
   - 确保 `import` 有序、无冗余，类结构符合标准（成员变量 → 构造器 → 方法）。
6. **禁止任何逻辑改动**  
   - 不得抽取新方法、不得内联、不得修改条件判断、循环、异常处理等任何执行逻辑。  
   - 所有优化仅限于注释、命名、格式层面，保持原始行为完全一致。

```
