# 一、**环境搭建与IDEA体验无损迁移**

## 1、核心认知转变：从 `IDEA` 到 `Cursor`

> 传统的IDEA是“规则驱动”的工具，依赖强大的静态代码分析；而 `Cursor` 是基于 `VS` `Code` 内核的“`AI`原生”编辑器。两者在 `Java`  生态中各有所长，最佳实践是组合使用或平滑过渡。



**核心建议：** 现阶段推荐“双修”。利用 `Cursor` 强大的AI生成能力和上下文理解来写代码，遇到复杂的断点调试、深度的依赖冲突排查时，一键切回IDEA处理。

| 维度       | IDEA + AI 插件 (Copilot/Assistant)          | Cursor (AI Native Editor)                          |
| ---------- | ------------------------------------------- | -------------------------------------------------- |
| 交互模式   | 补全/问答式，被动响应（受限于 IDE 沙箱）    | 深度内联编辑 + `Composer` 跨文件主动编排           |
| 上下文感知 | 依赖 `IDE` 本地索引，插件获取上下文有割裂感 | 原生向量数据库索引 (`@codebase`)，全局语义无缝融合 |
| 能力边界   | 仅能在当前文件或对话框内生成/修改代码       | 可同时创建、修改、删除多个文件并直接应用 `Diff`    |
| 工作流位置 | 编码辅助环节（写代码时的“外挂”助手）        | 贯穿需求分析→架构设计→批量编码→测试全流程          |
| 思维模型   | “帮我把这段 Java 代码写完/解释一下”         | “帮我实现这个功能模块，自动处理好所有文件引用”     |

- **交互模式的区别**：
  - `IDEA` 的 `AI` 插件本质上是“寄生”在强大的传统 `IDE` 之上的，它必须遵循 `JetBrains` 的安全沙箱规则，因此很难做到跨文件的自动化修改。
  - 而 `Cursor` 是从底层重构了编辑器内核，它的 `Composer` (`Ctrl+I`) 就像一个拥有项目读写权限的工程队，能够真正执行多文件联动的复杂指令。
- **上下文感知的深度**：
  - `IDEA` 本身拥有极强的静态代码分析能力（索引），但 `AI` 插件往往无法无损地实时调用这些深层数据。`Cursor`  则自带了一套为大模型优化的向量检索系统，
  - 当你使用 `@codebase` 时，它能比传统插件更精准地在海量代码中定位到相关联的逻辑片段。
- **工作流与思维模型**：
  - 使用 `IDEA` 插件时，你的思维依然是传统的“逐行/逐文件编码”，`AI` 只是帮你省去了打字的力气。
  - 在 `Cursor` 中，你可以切换到“指挥官模式”，直接下达业务目标（如“新增一个积分签到模块”），让它一次性搞定 `Entity`、`Mapper`、Service、`Controller` 的全套生成与关联，彻底改变了开发的心智负担。



## 2、`Cursor` 核心快捷键速查表

| 快捷键     | 功能名称     | 核心作用与区别                                               |
| ---------- | ------------ | ------------------------------------------------------------ |
| `Tab`      | 智能代码补全 | 【被动触发】 `AI` 预测你的下一步操作，按` Tab` 直接采纳灰色建议。 |
| `Ctrl + K` | 内联编辑     | 【原地修改】 在当前光标处或选中代码上，弹出一个小的输入框。适合小范围、精准的快速修改。 |
| `Ctrl + L` | 侧边栏聊天   | 【对话交流】 唤起右侧的对话框。适合解释代码、提问、生成新文件或进行多轮探讨，不会直接改动你的原代码。 |
| `Ctrl + I` | `Composer`   | 【跨文件执行】 唤起一个强大的全能输入框。适合牵一发而动全身的大需求，能同时创建和修改多个文件。 |





# 二、核心交互与上下文精准控制

## 1、智能代码补全（`Cursor` `Tab`）

在深入复杂的对话之前，先感受一下最直接的提效工具。`Cursor` `Tab` 超越了传统 `IDE` 基于语法的单行补全，它能根据你的项目逻辑进行**多行甚至整段函数的预测**。

- **日常使用**：就像你在 `IDEA` 里按 `Tab` 一样自然，但在处理重复性样板代码（如 `Java` 实体类的 `Getter/Setter`、简单的 `DTO` 转换）时，它的预判会非常精准。
- **进阶技巧**：当你写下一行注释（例如 `// 校验用户手机号格式并查询数据库`），按下 `Tab` 键，`Cursor` 往往能直接生成包含正则校验和`MyBatis-Plus` 查询的完整逻辑块。





## 2、上下文引用系统（@符号）：给 AI 指路

> 这是 `Cursor`的灵魂功能。大模型没有记忆，每次对话都需要你通过 `@` 符号把相关的代码、文档投喂给它。对于 `Java` 这种强依赖上下文的语言，用对 `@` 至关重要。

| 符号        | 作用说明                                         | Java实战场景举例                                             |
| ----------- | ------------------------------------------------ | ------------------------------------------------------------ |
| @文件名     | 将指定文件的完整内容注入上下文                   | 帮我优化 `@UserService.java`里的这段业务逻辑                 |
| @文件夹名   | 注入整个目录的结构信息                           | 分析  `@src/main/java/com/example/controller` 下的接口规范   |
| `@codebase` | 触发语义搜索，让 `AI` 自己在整个项目中找相关代码 | `@codebase` 找出所有调用了 `OrderMapper` 的地方              |
| `@docs`     | 引入已索引的第三方官方文档                       | 参考 `@SpringBoot Docs `帮我配置一个自定义的 `Starter`       |
| `@web`      | 触发实时网络搜索                                 | `@web` 帮我查一下 `Spring Boot 3.2` 版本最新的安全漏洞修复方案 |
| `@git`      | 引用 `Git` 历史提交记录或 `Diff`                 | 查看 `@git` 昨天提交的改动，解释一下为什么要这么改           |



### 1）**场景一：我想优化某个具体的类（使用 `@`文件名）**

- **错误做法**：直接把 `UserService.java` 的代码全选复制粘贴到对话框里问 AI 怎么优化。
- **正确做法**：按下 `Ctrl+L` 打开聊天框，输入 `@UserService.java 请帮我检查这个类里有没有潜在的空指针异常，并优化代码结构`9。AI 会自动读取该文件的最新内容进行分析。



### 2）**场景二：我不知道代码在哪，想全局查找（使用 `@codebase`）**

- **实战**：你想找出项目里所有用到 `Redis` 的地方。直接在聊天框输入：`@codebase 帮我列出项目中所有使用了 RedisTemplate 的地方，并说明它们是用来做什么的`。`AI` 会扫描整个项目索引，给你一份汇总报告。



### 3）**场景三：我想查第三方框架的用法（使用 `@docs`）**

- **实战**：你忘记了 `Spring Boot 3.x` 中配置跨域的新写法。输入：`@Spring Boot Docs 如何在 Spring Boot 3 中使用 WebMvcConfigurer 配置跨域？请给出最新的 Java Config 代码示例`。`AI` 会去查阅官方文档并给你标准答案。





## 3、对话式编程：告别模糊的大白话

> 在日常聊天框（Cmd/Ctrl + L）中与 `AI`交互时，作为资深开发，你需要学会用**结构化的指令**来约束AI的输出，避免它生成“能用但不好用”的代码。

| 核心要素 | 说明                     | Java开发示例                                                 |
| -------- | ------------------------ | ------------------------------------------------------------ |
| 目标     | 要实现的具体功能         | 在 `OrderService` 中新增一个批量取消订单的方法               |
| 约束     | 代码规范、技术栈限制     | 必须使用 `MyBatis-Plus` 的 `updateBatchById`；遵循阿里巴巴 `Java`开发手册；添加详细的中文注释 |
| 输出格式 | 期望的代码结构和附加说明 | 给出完整的 `Service` 接口和实现类代码；附带一个 `JUnit 5` 的单元测试用例 |
| 验证要求 | 让 `AI`自检代码逻辑      | 检查是否有事务遗漏，确认批量操作的内存占用是否合理           |



## 4、跨文件协同编辑 `Composer`：全局架构调整神器

> 如果说 `Cursor` `Chat` 适合单点突破，那么 **`Composer` (Cmd/Ctrl + I)** 就是为你准备的“多文件批量手术刀”。当你的需求涉及 `Controller`、Service、DTO 等多个文件的联动修改时，`Composer` 极其强大。

- **典型场景**：你需要给现有的订单系统增加一个“发货地址快照”字段。
- **操作流程**：
  1. 打开 `Composer`  (`Cmd`/`Ctrl + I`)。
  2. 输入指令：“为订单模块增加收货地址快照功能。需要修改 `@OrderEntity.java` 增加字段，更新 `@OrderDTO.java`，并在 `@OrderService.java` 中完善保存逻辑。”
  3. `Cursor` 会自动分析依赖关系，同时在多个文件中生成或修改代码，并以 `Diff`（差异对比）的形式呈现给你。
  4. 你只需要一键 `Accept` `All`（全部接受）即可



## 5、团队规范固化：利用 `.cursorrules` 统一代码风格

> 这是本阶段的终极武器。在日常开发中，我们经常会遇到 AI 生成的代码不符合团队规范的情况（比如没用指定的异常处理类，或者日志格式不对）。

你可以在项目根目录下创建一个名为 `.cursorrules` 的文件。这个文件里的内容会被自动注入到每一次与 `AI` 的对话中，相当于给 `AI` 立下了**永久的“家规”**。

```java
你是一名资深的Java后端开发工程师。在生成代码时，请严格遵守以下规范：

1. **技术栈约束**：
   - 始终使用 Java 17 的语法特性（如 record, switch expression）。
   - ORM 框架统一使用 MyBatis-Plus，查询尽量使用 LambdaQueryWrapper。
   - 对象转换统一使用 MapStruct，禁止使用 BeanUtils.copyProperties。

2. **代码风格**：
   - 所有的接口返回值必须封装在 `Result<T>` 统一响应体中。
   - 业务异常必须抛出 `BusinessException`，并携带明确的错误码。
   - 日志打印必须使用 Slf4j，且必须包含 traceId。

3. **安全与性能**：
   - 严禁在循环中进行数据库查询。
   - 涉及金额计算的字段，必须使用 BigDecimal。
```







# 三、Java企业级项目全流程实战

## 1、需求分析与架构设计：让 `AI` 充当你的“技术参谋”

- **实战场景**：产品经理提了个模糊需求——“我们要加一个用户积分签到功能”。你不想直接上手写，想先理清思路。

- **怎么操作**：按下 `Ctrl+I` (`Composer`)，输入以下指令：

  > 我要开发一个用户每日签到领积分的功能。请基于 Spring Boot + Redis 帮我出一份技术方案草稿。
  >
  > 要求：
  >
  > 1. 给出 `MySQL` 的建表语句（包含用户ID、签到日期、连续签到天数）；
  > 2. 说明如何利用 `Redis` 的 `Bitmap` 或 `Set` 结构来高效记录签到状态；
  > 3. 生成 `Mermaid` 格式的时序图代码，展示从 `Controller` 到 `Service` 再到 `Redis/DB` 的交互流程。”

  **输出：**`Cursor` 会立刻给你返回完整的 `SQL`、`Redis` 选型分析以及可以直接渲染成图的 `Mermaid` 代码，你拿着这份方案去跟团队对齐，既专业又高效。



## 2、核心业务代码生成：分层开发的自动化流水线

- **实战场景**：你需要为刚才的“积分签到”功能写具体的 `CRUD` 代码。

- **怎么操作**：继续在 `Composer` (`Ctrl+I`) 中输入，并带上上下文：

  > 参考项目里现有的 `@BaseController.java` 和 `@Result.java` 规范，帮我生成积分签到的全套代码。
  >
  > 1. 创建 `CheckInEntity` (对应刚生成的表)、`CheckInDTO`、`CheckInVO`。
  > 2. 在 `CheckInService` 中实现 `doCheckIn(Long userId)` 方法：逻辑是先查 `Redis` 今天是否已签到，未签到则写入 `Redis` 并异步落库 `MySQL`，同时增加连续签到天数。
  > 3. 在 `CheckInController` 中暴露 `/api/checkin` 接口。
  >
  > 注意：使用 `MyBatis-Plus`，加上 `Lombok` 注解，关键逻辑要加中文注释。”

  **输出：**几秒钟后，你会发现 Cursor 自动在你的项目目录里创建了 Entity、DTO、VO、Service、Controller 等多个文件，并且代码风格和你现有的项目高度一致。你只需要点一下“Accept All”，基础代码就全部就位了



## 3、自动化测试与调试：补齐被遗忘的单元测试

- **实战场景**：代码写完了，但你还得补单元测试才能合并分支。手写 `Mock` 数据太繁琐。

- **怎么操作**：打开刚刚生成的 `CheckInServiceImpl.java`，选中那个核心的 `doCheckIn` 方法。按下 `Ctrl+K`，输入指令：

  > 为这个选中的方法生成一个完整的` JUnit 5` 单元测试。要求：使用 `Mockito` 模拟 `CheckInMapper` 和 `StringRedisTemplate`；覆盖‘首次签到’和‘重复签到’两种情况；断言返回值是否符合预期。

  输出：`Cursor` 会在当前目录下自动生成一个 `CheckInServiceImplTest.java` 文件，里面包含了配置好的 `@Mock`、`@InjectMocks` 以及带有详细断言的测试用例。你甚至不需要改什么，直接右键 `Run` 就能跑通。



## 4、遗留代码重构：给“屎山”代码做微创手术

**实战场景**：你在老项目中看到了这样一段让人头皮发麻的代码（典型的支付渠道判断）：

```java
public void pay(String channel, BigDecimal amount) {
    if (channel.equals("ALIPAY")) {
        // 支付宝支付逻辑...
        System.out.println("使用支付宝支付了：" + amount);
    } else if (channel.equals("WECHAT")) {
        // 微信支付逻辑...
        System.out.println("使用微信支付了：" + amount);
    } else if (channel.equals("UNIONPAY")) {
        // 银联支付逻辑...
        System.out.println("使用银联支付了：" + amount);
    }
    // ...可能还有几十个else if
}
```

- **怎么操作**：选中这段代码，按下 `Ctrl+K`（或者在 Chat 中用 `@文件名` 引用），输入指令：

  > 这段代码违反了开闭原则，且圈复杂度太高。请使用**策略模式 + 工厂模式**对其进行重构。
  >
  > 要求：定义一个 `PaymentStrategy` 接口，将支付宝、微信等逻辑抽离成独立的实现类，并通过一个 Map 容器来动态获取对应的策略，消除所有的 ·if-else·。

  **输出：**`Cursor` 会帮你把这一坨代码拆解成：`PaymentStrategy` 接口、`AlipayStrategy`、`WechatStrategy` 等实现类，以及一个负责管理策略的 `PaymentContext`。原本几十行的混乱代码，瞬间变成了清晰、易扩展的标准设计模式代码。
