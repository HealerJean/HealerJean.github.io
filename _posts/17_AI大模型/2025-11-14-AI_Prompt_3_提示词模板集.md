## 1、后端

### 1）单元测试

```markdown
## 背景信息

作为 Java 架构师，需为以下类或方法生成对应的单元测试：
- 单侧类（目标测试类目录<单侧类已存在，进行编辑即可>：src/test/impl）
   - com.demo1
   - com.demo2
- 单侧类（目标测试类目录：src/test/im2）
   - com.demo3


## 要求与规范

### 1. 命名规范
- 所有测试方法必须使用驼峰命名法（如 `testExportSuccessWithValidData`），禁止使用下划线。

### 2. 测试框架与注解
- 测试类必须使用 `@ExtendWith(MockitoExtension.class)` 作为运行器。
- 被测服务类使用 `@InjectMocks` 注解注入。
- 所有**外部依赖**（如 Service、Repository、Util 等）使用 `@Mock` 注解模拟。
- 每个测试方法必须使用 `@DisplayName` 注解，并用**中文明确描述测试场景**（例如：`@DisplayName("导出成功 - 输入有效业务数据")`）。

### 3. 数据与 Mock 管理
- 所有 Mock 行为和测试数据必须在**每个测试方法内部定义**，禁止使用 `@BeforeEach`、`@Before` 或任何全局初始化方法。
- 必须覆盖以下逻辑分支：
  - 正常成功路径（含典型业务数据）
  - 异常路径（如服务抛出业务异常、系统异常）
  - 边界条件（如空输入、空集合、单元素集合、超大集合）
  - 条件分支（每个 `if/else`、`switch` 分支）
  - 异常处理（每个 `try/catch` 块的 catch 路径）
- **不得 Mock 被测类自身的任何方法**（仅允许 Mock 其依赖的外部组件）。
- **所有 Mock 的对象和集合类（如 `List`、`Set`、`Map`）必须进行赋值，不能 Mock 空对象、空集合

### 4. 依赖合法性
- 生成的代码中引用的所有类、字段、方法必须确保在项目当前模块或 Maven 依赖中真实存在，禁止引入不存在的类、字段、过时 API。
- 每个测试方法上方必须包含独立注释块，说明：
  - 测试目的
  - 覆盖的具体逻辑分支
  - 预期行为（如返回值、异常类型、调用次数等）
- 禁止使用行尾注释（即 `// 注释` 写在代码行末），所有注释必须独立成行并置于相关代码上方。

### 5. 集合初始化规范
- 所有集合（如 `List`、`Set`、`Map`）初始化时必须显式指定**最小合理初始容量**（例如：已知将添加 3 个元素，则使用 `new ArrayList<>(3)`；若集合为空，则使用 `new ArrayList<>(0)`）。

### 6. 其他限制
- 仅允许创建上述类对应的单元测试类（文件名需符合 JUnit 规范，如 `DepositInvoiceDetailExportHandlerTest.java`）。
- 不得修改任何业务逻辑代码。
- 不得额外创建工具类、配置类、辅助类或测试基类。
```

### 2）代码注释

```markdown
请对以下 Java 代码进行深度美化与注释增强，严格遵循以下 6 点要求：

1. 优化类、方法、成员变量的注释
   - 为类添加完整 Javadoc 注释，说明核心职责、所属业务模块、关键行为。
   - 为所有 public/protected 方法添加完整 Javadoc，包含 @param、@return、@throws（如适用），并说明业务语义与使用场景。
   - 成员变量上方添加业务含义注释，禁止使用行尾 // 注释，不允许无意义重复注释。

2. 方法内部必须使用【两级结构化层级注释】（重点要求）
   - 方法外部 Javadoc 中用 1、2、3 概括整体执行步骤（方法总流程）。
   - 方法内部使用：
     // 1. 外层大步骤（准备/处理/返回）
     //    1) 内层子步骤
     //    2) 内层子步骤
   仅允许两级注释，不允许更多层级，逻辑分段必须清晰。

3. 优化所有变量命名，提高可读性
   - 禁止使用 data、obj、tmp、res、flag、result 等模糊词汇。
   - 命名必须具备业务语义，优先使用名词短语或动宾结构，做到见名知意。

4. 代码格式规范化
   - 统一缩进、换行、大括号风格，修复格式混乱问题。
   - 魔法值必须提取为常量，去除冗余 import，保持类结构标准：成员变量 → 构造器 → 方法。

5. 严格禁止任何逻辑改动
   - 不得修改业务逻辑、条件判断、循环、异常处理。
   - 不得抽取新方法、不得内联代码、不得改变原有执行流程。
   - 仅允许优化注释、命名、格式、结构排版。

6. 整体风格要求
   - 注释必须完整、规范、无冗余、不啰嗦。
   - 代码结构必须体现：准备 → 处理 → 验证/返回 标准流程。
   - 最终代码必须达到企业级 Java 开发规范，可直接上线使用。
```
